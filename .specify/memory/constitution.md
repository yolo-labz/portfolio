# Portfolio Monorepo Constitution

## Core Principles

### I. Sell, Don't Tell

Every element of this portfolio MUST serve a sales function. The audience is
Upwork clients evaluating whether to hire Pedro for data extraction, automation,
or full-stack work.

- Lead with measurable outcomes (throughput, uptime, cost savings), not
  self-descriptions or buzzword lists.
- Project pages MUST include: a concrete problem statement, the solution
  delivered, and quantified results.
- Skills sections organize by domain (what problems they solve), not by
  technology name alone.
- The hero section presents impact metrics, not a greeting.

### II. Static-First Architecture

All portfolio content is statically generated at build time. No CMS, no runtime
data fetching, no client-side API calls for content.

- Project metadata, skills, and experience data live in typed TypeScript
  constants under `apps/web/src/data/`.
- Content changes are code changes â€” they go through version control, linting,
  and CI.
- Dynamic project pages use `generateStaticParams` from the data layer.
- This ensures sub-second load times and zero hosting cost on static platforms.

### III. Monorepo Discipline

The Turborepo + pnpm workspace structure MUST be maintained. Each workspace
member has a clear boundary.

- `apps/web` is the only deployable application. It depends on `@portfolio/ui`.
- `packages/ui` exports raw TypeScript source. No build step. Next.js transpiles
  it via `transpilePackages`.
- `projects/*` have thin `package.json` wrappers for Turbo orchestration only.
  Their real tooling (Python, Terraform, Docker) is self-contained.
- Root-level `pnpm lint`, `pnpm typecheck`, and `pnpm build` MUST pass before
  any merge to `main`.

### IV. Visual Polish Over Feature Count

This portfolio sells on craft, not feature breadth. A polished five-page site
beats a sprawling half-finished one.

- Animations MUST respect `prefers-reduced-motion` via the `useReducedMotion`
  hook. No gratuitous motion.
- The oklch color system and dark theme are intentional design decisions â€” do not
  introduce a light theme unless explicitly scoped.
- Typography uses Inter (sans) and JetBrains Mono (mono). No additional fonts
  without design justification.
- Lighthouse targets: 95+ on Performance, Accessibility, Best Practices, and
  SEO.

### V. No AI Slop

All copy MUST read as if written by a working engineer, not generated by an LLM.

- Banned phrases: "passionate developer", "innovative solutions", "cutting-edge
  technology", "leveraging the power of", "synergy", "drive impactful results".
- Use "data extraction" instead of "scraping" in any Upwork-facing content.
  Upwork flags scraping language and has rejected Pedro's project catalog for it
  before.
- Descriptions MUST be specific. "Built data ingestion pipeline processing 5TB+
  daily" is good. "Experienced with big data" is not.
- No filler sections. If a section doesn't help close a deal, remove it.

### VI. Mixed-Stack Autonomy

Projects under `projects/*` span Python, Terraform, Docker, and JavaScript.
Each project manages its own language tooling independently.

- Python projects use `pyproject.toml` and their own virtual environments. There
  is no root-level Python environment.
- Biome ignores `*.py` files â€” Python linting uses ruff within each project.
- Turbo tasks in project stubs delegate to the appropriate toolchain via shell
  scripts in their `package.json`.
- The `.python-version` file at the repo root sets 3.12 for pyenv consistency.

## Deployment & Hosting

- The portfolio site (`apps/web`) deploys as a static export or via Vercel/
  Cloudflare Pages.
- GitHub Actions CI enforces lint â†’ typecheck â†’ build on every push and PR to
  `main`. Merging with a broken pipeline is not permitted.
- Individual projects may have their own deployment infrastructure (Docker
  Compose, Terraform, GitHub Actions) that is independent of the portfolio site
  pipeline.

## Content & Copy Standards

- Project data schema is defined in `apps/web/src/data/projects.ts` via the
  `Project` interface. All fields are required except optional links.
- Each project MUST have: `slug`, `title`, `tagline`, `description`,
  `problemStatement`, `solutionSummary`, `techStack`, `metrics`, `gradient`.
- Metrics MUST be concrete and verifiable (or clearly aspirational for demo
  projects). No vague qualifiers like "fast" or "scalable" without numbers.
- Tech stack items MUST specify a `category` (language, framework, database,
  infrastructure, service) for consistent rendering.

## Governance

This constitution governs all development decisions for the portfolio monorepo.
It supersedes ad-hoc preferences and prior conversation context.

- Amendments require updating this file, incrementing the version, and updating
  the Sync Impact Report comment at the top.
- Version follows semver: MAJOR for principle removal/redefinition, MINOR for
  new principles or sections, PATCH for wording/clarification.
- All PRs SHOULD be reviewed against these principles before merge.
- The CLAUDE.md file at the repo root provides operational guidance consistent
  with this constitution. If they conflict, this constitution takes precedence.
